# Selected Measures Lookup

Use this file for vectorless RAG measure retrieval and dependency backtracking.

## How to use this file

When the user asks for a measure or KPI:

1. Search the `measure_name` column for the closest or exact measure.
2. Prefer exact match over partial match.
3. If exact match is not found, search the `normalized_name` column.
4. Read the `depends_on` column.
5. For every measure listed in `depends_on`, search that measure again in this same table.
6. Repeat recursively until the measure has no dependencies.
7. Only after resolving the full dependency chain, generate SQL.
8. Use `finance_context.md` for SQL joins, EWIG/community filters, MainAccountCategory_Description matching, GL hierarchy rules, Real Estate / Leasing / GL rental revenue routing (`Unit_Unfiltered`, `UnitLeasingWithLocation`, `GLTransactions` building/unit columns), Payables State Transactions / DueAnalysis rules, date rules, and formatting rules.

Important:

- For Payables State Transactions measures, remember that Payables Balance is a latest-date snapshot measure. Average Payables is an average of daily balances. Payables Before Due uses `DueAnalysis.Group1 = -1`; Payables Overdue / After Due uses `DueAnalysis.Group1 = 1`.
- For **payables due analysis by buckets** (AP aging on payables — not generic “due analysis”), use `DueAnalysis.DueDays` (`da.DueDays`) and the Due Overdue bucket SQL/layout in `finance_context.md`; do not use Group1/Group2/Group3 for bucket mapping for now.
- Do not generate SQL from `[X] - [Y]` until X and Y are resolved.
- Do not treat every `[Column Name]` as a measure. Use the `depends_on` column for measure dependencies.
- If a measure has no dependencies, treat it as a base/direct measure and map it to SQL using `finance_context.md`.
- Percentage measures must be normalized using the word `percent`, not by removing `%`.
- For business/report terms, check MainAccountCategory_Description matching rules in `finance_context.md` before applying L1-L7 hierarchy filters. Do not use `GLAccount.AccountCategory` for now.
- For units, lease expiry, rent potential, or rental growth: follow **Real Estate / Leasing / GL Rental Revenue Rules** in `finance_context.md` (do not use `UnitWithRM`; do not mix rent potential with posted GL rental revenue).
- For vacancy aging, re-let time, occupancy vs portfolio average, or tenant demand by area: follow **Vacancy, Re-let, Occupancy, and Tenant Demand Rules** in `finance_context.md` (`Unit_Unfiltered` / `UnitLeasingWithLocation`; not `GLTransactions`).
- For average tenant stay, top-revenue tenants (net sales), or tenant retention/renewal trend: follow **Tenant Stay, Tenant Revenue, and Tenant Retention Rules** in `finance_context.md`.
- For portfolio rental income YTD, rental yield (highest/lowest by building), top-building revenue share, or asset-class revenue: follow **GL Rental Income, Rental Yield, Building Contribution, and Asset Class Revenue Rules** in `finance_context.md`.
